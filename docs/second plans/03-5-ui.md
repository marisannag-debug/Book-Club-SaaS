# Plan wdrożenia UI — Stage 0 (zadania 1–3)

## 1. Krótkie podsumowanie celu i priorytetów
- Cel: dostarczyć spójne, rozszerzalne UI dla minimalnego end‑to‑end flow: scaffold projektu, core DB oraz Clubs CRUD + strona tworzenia klubu.
- Priorytety:
  - 1) Stabilne kontrakty API i typy (kompatybilność z przyszłymi polami: `inviteCode`, `organizerId`).
  - 2) Prostota implementacji formularza tworzenia klubu (klient+serwer walidacja).
  - 3) Testowalność (unit/integration/E2E) i łatwe rozszerzanie na etapy 4–9.

UI jest fundamentem dla invite/members/proposals/votes/chat/meetings — musi zapewnić przewidywalne kontrakty i miejsca rozszerzeń (component slots, optional fields, feature flags).

## 3. Architektura komponentów i struktura katalogów
- `components/ui/` — thin wrappers nad shadcn/Tailwind (ARIA defaults, className merging).
- `components/clubs/` — domain components (ClubForm, ClubCard, ClubsList). Są bardziej stateful.
- Granularność: atomic (Input, Button) → molecules (FormField, Toast) → organisms (ClubForm) → pages (`pages/app/clubs/new.tsx`).
- Trzymać style i tokens w `styles/`, helpery UI w `lib/ui.ts` (merge classNames, theme helpers).

## 4. Typy i kontrakty danych (TypeScript)
```ts
export type Club = {
  id: string
  name: string
  description?: string
  organizerId?: string | null
  inviteCode?: string | null
}

// POST /api/clubs body
type CreateClubRequest = { name: string; description?: string }
// POST response
type CreateClubResponse = { id: string; inviteCode?: string }
// GET /api/clubs response
type ListClubsResponse = { clubs: Club[] }
```

- Wersjonowanie kontraktu: dodawaj nowe, opcjonalne pola (nullable) i aktualizuj `CreateClubResponseV2` jeśli konieczne. Serwer powinien dalej odpowiadać zgodnie z dotychczasową wersją.

## 5. Formularz tworzenia klubu — plan implementacji
- Biblioteki: `react-hook-form`, `@hookform/resolvers/zod`, `zod` (shared `lib/validators/club.ts`).
- UI: użyj `components/ui/*` wrapperów (Input, Textarea, Button, Card, Toast).
- UX: disable submit podczas request, pokaż loader w buttonie, podświetl pola z błędami, toast sukcesu/błędu, redirect na `/app/clubs/[id]` po 201.

Krótki komponent (przykład, ~10 linii):
```tsx
// components/clubs/ClubForm.tsx (fragment)
const ClubForm = ({ onCreated }) => {
  const { register, handleSubmit } = useForm({ resolver: zodResolver(ClubCreateSchema) })
  async function onSubmit(data){ const res = await fetch('/api/clubs',{method:'POST',body:JSON.stringify(data)}); const j=await res.json(); onCreated(j.id) }
  return (<form onSubmit={handleSubmit(onSubmit)}><Input {...register('name')} /><Textarea {...register('description')} /><Button type="submit">Utwórz</Button></form>)
}
```

Fetch snippet (error handling + redirect):
```ts
const res = await fetch('/api/clubs', { method: 'POST', headers:{'Content-Type':'application/json'}, body: JSON.stringify(payload) })
if (!res.ok) throw await res.json()
const { id } = await res.json()
router.push(`/app/clubs/${id}`)
```

Zod import usage (client):
```ts
import { ClubCreateSchema } from 'lib/validators/club'
const parsed = ClubCreateSchema.safeParse({ name, description })
```

## 6. Integracja z endpointami — praktyczne wskazówki
- GET list: paginacja (limit/offset) i proste prefetch na dashboard.
- POST create: idempotency — jeśli klient może wysłać wielokrotnie, serwer powinien wykorzystywać unikalny constraint lub klient może wysyłać `clientRequestId` w header by dedupe.
- Obsługa błędów: parsuj JSON z `issues` przy 400 i zamapuj na pola formularza.
- Retry: krótkie retry w kliencie (exponential backoff) tylko dla błędów 5xx; nie retryuj 4xx.

## 7. State management i fetching
- Rekomendacja: SWR (lekki) albo React Query. Propozycja: SWR dla list i pojedynczych fetchy.
- Po stworzeniu klubu: `mutate('/api/clubs')` aby odświeżyć listę.
- Optimistic updates: niezalecane przy create (nie znamy id), użyć pessimistic + show loading.
- SSR/CSR: listy klubów mogą być CSR z prefetch (prefetch na dashboard load), pojedyncza strona klubu SSR tylko gdy potrzebujesz SEO.

## 8. Dostępność i responsywność
- ARIA: pola formularza powinny mieć `aria-invalid` i `aria-describedby` dla błędów.
- Keyboard: focus order, `Enter` submit, `Escape` closes modals.
- Focus states: wyraźne outline (Tailwind ring utilities).
- Mobile-first: użyj prostych układów, single-column formularz, responsive paddings.

## 11. Acceptance criteria / definicje gotowości
- Manual:
  - Otwórz `/app/clubs/new`, wypełnij minimalne pola i kliknij Utwórz → redirect do `/app/clubs/:id`.
  - Sprawdź DB — nowy wpis w `clubs` z `invite_code`.
- Automatyczne:
  - Unit + integration tests green.
  - E2E create-club scenario green.

## 12. Integracja z przyszłymi etapami (kompatybilność)
- Open-for-extension kontrakt: nowe pola zawsze `optional` i `nullable`.
- Dodawanie pól DB: stosować `ALTER TABLE ADD COLUMN ... DEFAULT NULL` i wdrożyć klienta, który ignoruje nieznane pola.
- Feature flags: wprowadź prosty `NEXT_PUBLIC_FEATURE_FLAGS` lub middleware feature toggle na serwerze.
- API versioning: nie potrzebne od razu; użyj `Accept-Version` header lub prefixu `/api/v1/...` jeśli złamiesz kompatybilność.

## 15. Potencjalne ryzyka i mitigacje
- Race przy inviteCode collisions: dodać unique constraint i retry insert on conflict.
- XSS in descriptions: sanitize on render or escape; store raw but escape in UI.
- Flaky E2E: use `data-cy` selectors and stable waits (avoid arbitrary sleeps).
